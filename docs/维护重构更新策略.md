# 模块维护、重构与更新策略 (Hermes)

## 一、维护原则

### 每个模块有明确负责人

```
go-server/auth/         → 开发者 A   (认证+权限无小事)
go-server/logs/         → 开发者 B
go-server/issues/       → 开发者 B
go-server/projects/     → 开发者 A
go-server/qa/           → 开发者 D
go-server/sensors/      → 开发者 C
go-server/instruments/  → 开发者 C
py-agent/               → 开发者 D
web-ui/                 → 开发者 B + D
```

**负责人职责：**
- 该模块的 bug 修复第一响应人
- 模块的 API 变更需要负责人审批
- 每季度检查一次模块健康度（测试覆盖率、已知 bug 数、依赖过期）

### 模块间界限不可破

- 不允许跨模块直接调对方 DB 表
- 不允许在非自己模块里新增导出函数然后让别人依赖
- 共享代码放 `go-server/common/`，改 common 需要至少两人 review

## 二、重构策略

### 什么时候重构

| 触发条件 | 处理 |
|----------|------|
| **三个相似功能复制粘贴** | 提取到 common，写单测 |
| **API 响应 > 500ms** | 加索引、加缓存、考虑异步 |
| **单文件 > 500 行** | 拆文件，按职责分 |
| **测试跑不过** | 先修 bug，再重构，不要混在一次 commit |
| **依赖大版本更新** | 单独开分支，CI 全绿才能合 |

### 重构流程

```
1. 开 feat/refactor-<module> 分支
2. 先补测试（重构前覆盖率不能低于重构后）
3. 重构代码
4. go test ./... 全绿
5. PR → 负责人 review → 合并
```

### 重构红线

- ❌ 重构同时加新功能（拆成两个 PR）
- ❌ 重构时改 API 契约（破坏性变更走下面的废弃流程）
- ❌ 不写测试就重构

## 三、更新与发布策略

### 版本号（语义化）

```
v主版本.次版本.修订号

v1.2.3
 │  │  └── 修订号: bug 修复、文档更新
 │  └───── 次版本: 新功能，向后兼容
 └──────── 主版本: 破坏性变更
```

### 发布流程

```
develop 分支稳定 → 打 tag v0.1.0 → 部署测试环境
                                    │
                              1-2 天验证
                                    │
                              合并到 main → 打 release tag
                                    │
                              部署生产环境
```

**回滚：** 每次发布前打 `pre-release` tag，出问题直接 `docker-compose down && git checkout v0.1.0-pre && docker-compose up`。

### 数据库迁移规范

```
migrations/
├── 001_init.sql
├── 002_add_projects.sql
├── 003_add_experiment_runs.sql
└── 004_add_qa_logs.sql
```

**规则：**
- 序号递增，不修改已有的迁移文件
- 每次迁移必须可逆（提供 down.sql）
- 迁移前自动备份（CI 脚本里做）
- 生产环境迁移需要两个人在场

## 四、API 版本管理

### 路径版本（推荐）

```
/api/v1/logs/report       ← 当前版本
/api/v2/logs/report       ← 新版本，不兼容时启用
```

**规则：**
- v1 和 v2 同时运行至少 2 周，给前端和 Agent 切换时间
- v1 废弃时先标记 `Deprecated: true`，再 4 周后删除
- 前端调用带上版本号：`/api/v1/logs/report`

### 废弃流程

```
1. 新 API 上线 (v2)
2. 旧 API 标记 Deprecated (v1)，日志里记录谁还在调
3. 前端/Agent 切换到 v2
4. 确认 v1 零调用后，等待 4 周冷静期
5. 删除 v1 代码
```

## 五、依赖管理

### Go 依赖

```bash
go mod tidy           # 清理未使用的依赖
go mod verify         # 验证依赖完整性
```

- 依赖更新单独开 PR，CI 全绿才能合
- 不对依赖打 patch（风险不可控）
- `go.sum` 必须提交

### Python 依赖

```bash
pip freeze > requirements.txt    # 锁定版本
```

- LightAgent 版本升级前先在测试环境验证所有 prompt 模板
- DeepSeek API 版本变更可能影响 Agent 行为——升级后跑一遍日报解析回归测试

### 前端依赖

```bash
npm ci                # 严格按 lock 文件安装
npm audit             # 安全检查
```

## 六、监控与健康检查

每个模块提供 `/health` 端点：

```json
{
  "module": "logs",
  "status": "ok",
  "db": "connected",
  "last_migration": "004",
  "uptime_hours": 72
}
```

运维仪表盘汇总所有模块健康状态，异常时触发 ntfy 告警。

### 模块异常处理

| 异常 | 处理 |
|------|------|
| 单个模块崩溃 | Go 内 recover，不影响其他模块。崩溃模块返回 503 |
| DB 连接断开 | 自动重连，连续 3 次失败告警 |
| Agent 调用超时 | 超时 30s，降级为「人工录入模式」 |
| 传感器推送断流 | 5 分钟内恢复不告警，超过 5 分钟推送 ntfy |
| 磁盘 > 80% | 推送警告，自动清理过期备份 |

## 七、定期维护任务

| 频率 | 任务 | 负责人 |
|------|------|--------|
| 每日 | CI 自动：go test + npm test + 备份验证 | 自动 |
| 每周 | 检查依赖安全漏洞 (npm audit / go vulncheck) | 轮值 |
| 每月 | 恢复演练（从备份恢复到临时库） | 轮值 |
| 每季度 | 模块健康度检查 + 清理 Deprecated API + 降采样传感器历史 | 各负责人 |
| 每半年 | 全量回归测试 + 文档更新 + 性能压测 | 全员 |
