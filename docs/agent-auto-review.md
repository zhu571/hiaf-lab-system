# Agent 自动入库三级置信度方案评估

## 结论

Hermes 的三级置信度方案比“所有解析结果默认草稿”更积极，也更接近实际可用系统；但它只有在一个前提下才合理：**高置信不能由 Agent 自己一句 `confidence=high` 决定，而必须由后端基于结构化规则、历史模式、权限和审计约束二次判定**。

如果只是让 Agent 输出置信度，然后高置信直接入库，这个方案比全量草稿更差，因为它把最关键的安全边界交给了最不稳定的一环。如果置信度是“Agent 建议 + Go 后端规则校验 + 用户可配置偏好”的组合判断，那么它比全量草稿更好。

## 1. 相比“全量草稿”更好还是更差

我建议采用三级方案，但要把它改成“规则门控的三级方案”。

相比全量草稿，它的优点是：

- 降低低风险、重复性日报录入成本。实验室日报往往有大量格式稳定的内容，例如固定项目、固定设备、固定参数记录，全部人工确认会很快变成负担。
- 更符合用户心理预期。用户愿意让系统自动处理显然正确的内容，但希望模糊内容停下来问人。
- 能把人工注意力集中在真正需要判断的记录上，而不是每条都点一次确认。

相比全量草稿，它的风险是：

- 一旦高置信误判，错误会直接进入正式记录，后续周报、经验库、统计分析可能继续引用这个错误。
- 用户看到“已录入”后通常不会复查，自动入库会放大误解析的影响。
- Agent 可能对自己生成的内容过度自信，所以不能把置信度当作事实。

因此最终判断是：**三级方案更好，但必须保留“默认草稿、可开启高置信自动入库”的产品策略。** 新用户、早期系统、未验证项目应默认全量确认；只有当用户主动开启、且规则命中高置信时，才允许自动入库。

## 2. 高置信判断标准

高置信应同时满足以下规则，缺一项就降为中置信或低置信。

### 输入来源规则

- 用户输入是普通日志描述，不是命令、审批、权限变更、仪器控制、删除、导出、发布经验等动作。
- 输入不包含明显 prompt injection 或越权指令，例如“忽略之前规则”“直接写数据库”“替我确认所有内容”“删除旧记录”。
- 如果来自 OCR 或图片，OCR 只能作为候选来源，不能单独触发高置信自动入库，除非用户同时提供明确文字说明。

### 日期规则

- 日期必须明确，或能由会话上下文唯一确定为今天。
- “昨天”“上周五”“刚才”等相对日期必须由后端转换成具体日期，并在用户当前时区下无歧义。
- 如果日期缺失、跨日、和最近一条记录冲突，降为中置信。
- 如果解析出的日期超出合理窗口，例如未来日期、过早历史日期、和用户近期工作周期明显不符，降为中置信或低置信。

### 项目与人员规则

- 项目必须能唯一匹配到用户有权限写入的项目。
- 如果项目名有多个近似匹配，不能高置信。
- 记录归属人员必须是当前用户，或者是当前用户有明确代录权限的人员。
- Agent 不得自动创建新项目、新人员、新标签作为高置信入库条件。

### 设备与数据规则

- 设备名必须唯一匹配设备白名单或已有别名。
- 参数名、单位、取值范围必须匹配已知 schema 或设备历史模式。
- 数值必须通过基本合理性校验，例如温度、压力、电压、流量、时间长度不能明显越界。
- 单位换算必须确定，不能从上下文猜测。单位缺失但字段需要单位时，降为中置信。
- 同一条输入中多个参数不能互相矛盾，例如“升温到 80 C”和“温度保持 20 C”同时作为最终结果。

### 内容完整性规则

- 必填字段完整：日期、项目、分类、正文、操作者。
- 正文必须能回溯到用户原始输入，不能主要由 Agent 补写。
- Agent 只能规范化表达，不能新增实验结论、原因分析或未提供的数据。
- 如果有附件、OCR、传感器数据引用，正式记录必须保存来源引用。

### 重复与冲突规则

- 与最近记录做去重检查。高度相似且日期、项目、设备相同的内容不能直接入库，应提示可能重复。
- 与同一设备同一时间段的已有记录冲突时，降为中置信。
- 同一输入拆成多条日志时，除非边界非常清楚，否则不能高置信自动入库。

### 权限与类型规则

- 只允许普通日报自动入库。
- Issue、经验候选、仪器命令、报告导出、权限变更、数据删除、批量修改必须人工确认。
- 即使用户开启“高置信自动入库”，也不应覆盖这些类型限制。

### 置信度计算规则

建议不要只保存一个 Agent 置信度，而是保存三类结果：

- `agent_confidence`：Agent 自评，只能作为参考。
- `rule_confidence`：后端规则判定结果，决定是否允许自动入库。
- `final_decision`：最终处理结果，例如 `auto_committed`、`draft_pending_review`、`rejected`。

高置信自动入库必须满足：

- `agent_confidence = high`
- `rule_confidence = high`
- 用户开启高置信自动入库
- 记录类型在自动入库白名单内
- 所有关键信息可追溯到原始输入或可信系统数据

## 3. 风险最大的点与缓解

最大风险不是“低置信没理解”，而是**错误内容被高置信自动入库后变成可信历史**。

这类错误最危险，因为它不会停在一条日志里，后续可能继续影响：

- 周报总结。
- 经验库候选。
- 实验统计。
- 设备状态判断。
- 项目进度回顾。
- 后续 Agent 检索和推理。

缓解方式：

- 高置信判定由后端规则门控，Agent 不能自批。
- 所有自动入库记录加明显标记，例如 `created_by_agent=true`、`auto_committed=true`、`confidence_snapshot`。
- 自动入库后给用户短窗口撤销，例如 10 分钟内一键撤回为草稿。
- Web 顶部不仅提醒待确认，也应有“最近自动录入”入口，方便快速复查。
- 每条自动记录保留解析来源、规则命中、原始输入、模型版本、prompt/tool schema 版本。
- 自动入库只适用于普通日志，不扩展到经验、Issue、仪器命令、权限和报告发布。
- 建立抽样复核机制：系统早期或模型升级后，对高置信自动入库进行一定比例人工复核。
- 一旦用户编辑了自动入库记录，应把这次差异反馈给规则和提示词评估，而不是只改数据。

## 4. 改进建议

### 把“高置信自动”改成“高置信且可撤销自动”

不要只通知“已录入”，建议通知文案改成：

> 已自动录入 1 条，可撤销或编辑

这比“已录入”更符合实验记录场景，也能降低用户对误入库的焦虑。

### 默认策略分阶段开启

建议分三阶段：

1. 冷启动阶段：全部生成草稿。
2. 观察阶段：系统显示“本条本可自动入库”，但仍需确认，用来验证规则准确率。
3. 自动阶段：用户手动开启高置信自动入库。

这样可以在真实数据上校准规则，而不是一上线就相信置信度。

### 用户偏好要细分

“高置信自动 / 全部确认”太粗。建议至少支持：

- 全部确认。
- 仅文本日报高置信自动。
- 文本日报和已知模板高置信自动。
- 永不自动处理 OCR。
- 指定项目允许自动入库。
- 指定设备相关记录必须确认。

### 中置信草稿要做成审核队列

Web 顶部红点是对的，但还不够。建议统一进入“待审核中心”，并支持：

- 一键通过。
- 快速编辑。
- 批量通过同类型记录。
- 显示为什么不是高置信。
- 显示原始输入与解析结果差异。

### 低置信不一定直接退回

“没理解，请改写下”适合完全无法解析的情况。但如果能识别出部分字段，低置信也可以生成“不可入库草稿”，让用户补全，而不是完全丢弃。

建议低置信分两类：

- `needs_clarification`：能提问补齐，例如“这是哪个项目？”
- `rejected`：疑似 prompt injection、越权、危险动作或完全无法解析。

### 增加“解析证据”字段

每个结构化字段最好带来源：

```json
{
  "date": {
    "value": "2026-07-14",
    "source_text": "今天",
    "source_type": "user_text",
    "confidence": "high"
  },
  "device": {
    "value": "gas_cell",
    "source_text": "气室",
    "source_type": "known_alias",
    "confidence": "high"
  }
}
```

这会让审核、调试和后续追责都更清楚。

## 建议后的处理表

| 级别 | 判断标准 | 处理 |
|------|----------|------|
| 高置信 | Agent 高置信、后端规则高置信、字段完整、项目和设备唯一匹配、数值范围合理、无重复冲突、用户开启自动入库 | 自动入库，标记来源，通知可撤销 |
| 中置信 | 字段基本可解析，但日期、项目、单位、设备、参数或重复性存在不确定 | 生成草稿，进入待审核中心，支持一键通过和编辑 |
| 低置信 | 无法形成有效日志、疑似 prompt injection、越权动作、危险动作、严重幻觉或关键字段缺失 | 能补齐则追问，危险或无效则拒绝 |

## 最终建议

采用 Hermes 的三级方案，但不要把它理解成“Agent 输出高置信就直接入库”。正确版本应是：

> Agent 给候选置信度，Go 后端做规则置信度和权限校验，用户偏好决定是否允许高置信普通日志自动入库；所有其他类型默认人工确认。

这个版本比全量草稿更可用，也没有明显牺牲系统可信边界。
