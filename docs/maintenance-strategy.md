# 实验室日志系统维护、重构与更新策略

> 版本：v1  
> 日期：2026-07-14  
> 适用团队：3-4 人实验室日志系统开发维护小组  
> 技术栈：Go + chi、Python + LightAgent、Vue 3、PostgreSQL  
> 相关文档：`collab-guide.md`、`api-contract.md`、`project-design.md`、`permission-audit.md`、`instrument-security.md`、`agent-auto-review.md`

## 1. 维护目标

本策略用于约束系统上线后的日常维护、重构、版本发布、数据库变更、API 演进和回滚流程。目标不是制造复杂流程，而是让 3-4 人小团队在人员有限的情况下做到：

- 每个模块有人负责，出现问题能快速定位到人和边界。
- 变更可审查、可测试、可回滚。
- 数据库、API、权限和仪器控制等高风险变更有明确门槛。
- 小版本可以快速发布，大版本 breaking change 有计划、有通知、有迁移。
- 文档、测试、监控和备份跟随代码一起维护。

默认规则：

- 生产环境稳定性优先于功能速度。
- 所有线上数据结构变化必须走迁移文件，禁止手工改生产库。
- API 变化必须先更新 `api-contract.md`。
- 权限、仪器控制、Agent 自动写入、AI 问答的变更必须经过至少 1 人复核。
- 没有回滚方案的变更不能发布到生产环境。

## 2. 13 个模块的日常维护责任划分

小团队不建议为每个模块固定一个唯一负责人。推荐使用“主责 + 备份 + 共同复核”模式：每个模块有一个主责人，一个备份人；高风险模块必须跨角色复核。

### 2.1 角色建议

3-4 人团队建议按能力而不是职级分工：

| 角色 | 主要负责 | 至少备份 |
|------|----------|----------|
| 后端负责人 | M1、M2、M3、M4、M6、M12 | 前端或运维成员熟悉 API 和迁移流程 |
| Agent/Python 负责人 | M5、M9、M10 中的 Agent 和解析逻辑 | 后端负责人熟悉接口和门控 |
| 前端负责人 | M11，以及 M2-M6 的页面工作流 | 后端成员能处理紧急文案/配置修复 |
| 运维/仪器负责人 | M7、M8、M13 | 后端负责人能执行发布、回滚和备份恢复 |

如果团队只有 3 人，可以合并为：

- A：Go 后端、数据库、发布。
- B：Python Agent、AI 问答、经验库。
- C：Vue 前端、仪器/传感器页面、用户反馈。

但 M8 仪器控制、M12 数据库迁移、M1 鉴权相关变更仍必须至少 2 人看过。

### 2.2 日常维护职责

每个模块主责人需要保证：

- 模块 README 或设计说明保持可读。
- API、数据库字段、权限规则发生变化时同步更新文档。
- 每周查看模块错误日志、慢查询和用户反馈。
- 依赖升级、漏洞修复和测试失败不长期悬挂。
- 线上事故后补齐测试或告警，不只修复表面症状。

备份人需要保证：

- 能在主责人不在时处理紧急回滚、配置修复和基本排障。
- 至少每月参与一次该模块 PR review。
- 知道该模块的入口、数据表、关键 API 和监控位置。

## 3. 重构时机判断标准和流程

### 3.1 什么时候应该重构

满足以下任意条件，可以提出重构：

- 同一类 bug 在 1 个月内出现 2 次以上。
- 单个文件超过 500 行且同时包含 handler、service、repository、权限判断等多层职责。
- 修改一个模块时经常需要改 3 个以上无关模块。
- 单元测试难以编写，必须依赖完整系统才能验证一个普通规则。
- 查询或页面响应明显变慢，且瓶颈来自结构问题而不是单个索引缺失。
- 数据模型已经无法表达真实业务，例如项目、实验批次、权限边界混在备注字段里。
- 安全或权限边界不清晰，例如 Agent、AI 问答、仪器控制共享写入口。
- 依赖库已经停止维护，继续使用会影响安全或升级。

以下情况通常不应重构：

- 只是代码风格不喜欢。
- 没有用户价值，也没有降低维护风险。
- 业务规则还不稳定，抽象会过早固定错误模型。
- 可以通过小修、小测试、小文档解决的问题。

### 3.2 重构等级

| 等级 | 范围 | 示例 | 要求 |
|------|------|------|------|
| R1 小重构 | 单模块内部，不改 API 和数据库 | 拆函数、提 service、整理错误处理 | 1 人开发，1 人 review，跑相关测试 |
| R2 中重构 | 单模块边界变化，可能影响前端或 Agent | 日志模块拆出附件服务、Issue 状态机重写 | 提前写重构说明，更新 API 文档，补集成测试 |
| R3 大重构 | 跨模块、数据库或权限模型变化 | 引入项目维度、日志表迁移、仪器权限重做 | 单独里程碑，分阶段发布，必须有数据迁移和回滚方案 |

### 3.3 重构流程

1. 写一页重构提案，说明问题、目标、不做什么、影响模块、回滚方式。
2. 团队确认重构等级。R1 可直接开分支；R2/R3 先拆任务。
3. 先补行为测试或回归用例，避免重构时改坏现有行为。
4. 分小 PR 合并，不在一个 PR 中同时移动大量文件、改 API、改数据库和改前端。
5. 每个阶段保持系统可运行。R3 可以使用 feature flag 或兼容层。
6. 合并前确认文档、测试、迁移、监控和回滚脚本都已更新。
7. 发布后观察至少 1 个工作日；高风险重构观察 1 周。

### 3.4 重构验收标准

重构完成必须满足：

- 用户可见行为不变，除非变更已经写入发布说明。
- 原有测试通过，新增测试覆盖新的边界。
- 监控指标没有明显恶化。
- 旧入口如果被废弃，必须仍保留兼容期或明确迁移说明。
- 新结构能让下一次相关改动更小，而不是只换一种复杂度。

## 4. 版本号策略和发布流程

### 4.1 语义化版本

系统版本号采用：

```text
MAJOR.MINOR.PATCH
```

含义：

| 类型 | 触发条件 | 示例 |
|------|----------|------|
| MAJOR | breaking change、不可自动兼容的数据模型变化、权限模型重大调整 | `1.8.4 -> 2.0.0` |
| MINOR | 向后兼容的新功能、新 API、新页面、新模块能力 | `1.8.4 -> 1.9.0` |
| PATCH | bug 修复、文案、样式、小性能优化、兼容性修复 | `1.8.4 -> 1.8.5` |

预发布版本用于测试环境：

```text
1.9.0-rc.1
1.9.0-beta.2
2.0.0-alpha.1
```

### 4.2 模块版本与系统版本

小团队不单独维护 13 套模块版本。推荐：

- 整体系统使用一个版本号。
- API 使用独立路径版本，例如 `/api/v1`、`/api/v2`。
- 数据库迁移使用递增序号。
- 前端、后端、Agent 在一次发布中记录各自 commit hash。

### 4.3 发布分支

推荐分支：

```text
main       # 生产稳定版本
develop    # 日常集成
feature/*  # 功能开发
fix/*      # 普通修复
hotfix/*   # 生产紧急修复
release/*  # 发布候选
```

发布候选分支命名：

```text
release/v1.9.0
```

紧急修复分支命名：

```text
hotfix/v1.8.5-token-refresh
```

### 4.4 发布流程

普通发布：

1. 从 `develop` 切出 `release/vX.Y.Z`。
2. 冻结新功能，只接受 bug 修复、文档、迁移修正。
3. 更新版本号和 `CHANGELOG.md`。
4. 执行后端测试、Agent 测试、前端构建、迁移 dry-run。
5. 在测试环境部署 release candidate。
6. 使用发布检查表验证登录、项目、日志、Issue、Agent、AI 问答、仪器只读/控制权限。
7. 主责人确认备份已完成。
8. 合并到 `main`，打 tag：`vX.Y.Z`。
9. 部署生产。
10. 发布后观察错误率、慢查询、关键任务和用户反馈。
11. 将 release 分支合回 `develop`。

紧急发布：

1. 从 `main` 切 `hotfix/*`。
2. 只修复一个生产问题，不夹带重构和新功能。
3. 跑受影响模块测试和必要集成测试。
4. 更新 PATCH 版本和 hotfix 说明。
5. 合并 `main`，打 PATCH tag，部署。
6. 再合回 `develop`，避免修复丢失。

### 4.5 发布检查表

每次发布前至少确认：

- 数据库已备份。
- 所有迁移文件按顺序执行成功。
- 回滚脚本或回滚步骤可用。
- API 文档与实现一致。
- 权限相关测试通过。
- Agent 自动写入门控没有被绕过。
- 仪器控制命令白名单没有意外放宽。
- 前端构建成功，关键页面可打开。
- 环境变量和密钥没有写入仓库。

## 5. 数据库迁移规范

### 5.1 文件命名

迁移文件必须使用递增序号，禁止改写已经发布到生产的迁移文件。

推荐格式：

```text
migrations/
├── 0001_init.up.sql
├── 0001_init.down.sql
├── 0002_add_projects.up.sql
├── 0002_add_projects.down.sql
├── 0003_add_logs_project_id.up.sql
└── 0003_add_logs_project_id.down.sql
```

规则：

- 序号 4 位递增。
- 一个序号必须同时有 `up` 和 `down`。
- 文件名说明要短，使用小写英文和下划线。
- 已发布迁移不得重写，只能新增迁移修正。

### 5.2 可逆要求

每个迁移必须可逆，除非有明确理由。不可逆迁移必须在 PR 中标注，并由至少 2 人确认。

可逆迁移示例：

- 新增表：`down` 删除表。
- 新增列：`down` 删除列。
- 新增索引：`down` 删除索引。
- 拆表迁移：`down` 至少能把核心字段合回旧表。

不可逆迁移示例：

- 删除历史数据。
- 合并多个字段导致原始字段无法恢复。
- 批量脱敏或清洗覆盖原值。

不可逆迁移要求：

- 发布前完整备份。
- 写清恢复方式。
- 在测试环境用生产脱敏数据演练。
- 避免和功能发布混在一起。

### 5.3 迁移编写规范

迁移必须满足：

- 不依赖本地临时文件。
- 不使用未固定顺序的批量更新。
- 大表变更分阶段执行，避免长时间锁表。
- 新增 `NOT NULL` 字段要先加默认值或分两步回填。
- 新增外键前先清理脏数据。
- 新增索引前评估写入性能影响。
- 所有数据修复 SQL 要可重复审查，不把业务逻辑藏在手工操作里。

### 5.4 迁移流程

1. 开发者新增 `up/down` 文件。
2. 本地空库执行全部 `up`。
3. 本地从旧版本数据库执行增量 `up`。
4. 执行 `down` 回滚到上一版本。
5. 对涉及数据改写的迁移，使用测试数据做前后校验。
6. PR 中说明影响表、数据量、预计耗时、是否锁表、是否需要停机。
7. 发布前备份生产库。
8. 发布时先执行迁移，再启动依赖新 schema 的服务。
9. 发布后检查迁移版本表和关键查询。

### 5.5 备份规范

生产数据库备份要求：

- 每日自动备份一次，保留 30 天。
- 每周完整备份一次，保留 12 周。
- 每月归档备份一次，保留 12 个月。
- 每次发布前手动或自动创建发布前备份。
- 备份文件必须加密存储，访问权限最小化。
- 每季度至少演练一次从备份恢复。

备份命名建议：

```text
lab_daily_report_YYYYMMDD_HHMMSS_pre_v1.9.0.dump
lab_daily_report_YYYYMMDD_daily.dump
```

恢复演练必须记录：

- 备份文件名。
- 恢复耗时。
- 恢复到的数据库版本。
- 抽查的关键数据。
- 发现的问题和修复动作。

## 6. API 版本管理和废弃流程

### 6.1 API 版本路径

所有业务 API 使用路径版本：

```text
/api/v1/{module}/{resource}
/api/v2/{module}/{resource}
```

规则：

- `v1` 内只做向后兼容变更。
- breaking change 必须进入新版本路径。
- 同一生产环境可以同时提供 `v1` 和 `v2`。
- 内部服务调用也必须走版本化 API，不使用未声明私有接口。

### 6.2 向后兼容变更

以下变化通常不需要升 MAJOR，也不需要新 API 版本：

- 响应中新增可选字段。
- 请求中新增可选字段。
- 新增 endpoint。
- 新增枚举值，但旧客户端遇到未知值仍能显示或降级。
- 错误 `details` 增加字段。
- 性能优化、分页优化、不改变语义的排序稳定化。

### 6.3 API 废弃流程

废弃 API 必须经历：

1. 标记 deprecated：更新 `api-contract.md`，说明替代接口。
2. 响应头加入废弃信息：

```text
Deprecation: true
Sunset: 2026-10-31
Link: </api/v2/logs>; rel="successor-version"
```

3. 日志记录调用方、用户、调用频率。
4. 至少保留 2 个 MINOR 版本或 3 个月，取更长者。
5. 在前端、Agent、脚本全部迁移后才能删除。
6. 删除前发布说明中再次提醒。

### 6.4 错误码稳定性

错误码是 API 契约的一部分。已有错误码不能随意改名。

推荐格式：

```text
permission_denied
not_found
validation_failed
conflict
rate_limited
instrument_lease_required
agent_review_required
```

允许新增错误码，但必须更新文档和前端处理。

## 7. Go/Python/Vue 依赖管理策略

### 7.1 通用原则

- 依赖必须有明确用途，不能为了少量代码引入大型库。
- 所有依赖版本必须锁定。
- 安全补丁优先于功能升级。
- 大版本升级单独 PR，不和业务功能混在一起。
- 依赖升级必须跑对应语言测试和构建。
- 不使用无人维护、下载量极低且安全能力不明的核心依赖。

### 7.2 Go 依赖

Go 使用 `go.mod` 和 `go.sum` 管理。

规则：

- 新增依赖前先确认标准库或现有依赖无法解决。
- 使用 `go get module@version` 固定版本。
- 提交 `go.mod` 和 `go.sum`。
- 每月执行一次依赖审查：

```bash
go list -m -u all
govulncheck ./...
go test ./...
```

升级策略：

- PATCH/MINOR 升级可合并到普通维护 PR。
- MAJOR 升级必须单独 PR，说明破坏性变化和回归范围。
- Web 框架、认证、数据库驱动、迁移工具属于核心依赖，必须 2 人 review。

### 7.3 Python 依赖

Python Agent 推荐使用 `pyproject.toml` 加 lock 文件，或 `requirements.txt` 加哈希锁定。团队可二选一，但一个仓库内不要混用多套规则。

规则：

- 运行时依赖和开发依赖分开。
- Agent、OCR、LLM SDK 等依赖升级必须验证解析准确率和拒绝路径。
- 依赖升级后至少跑：

```bash
python -m pytest
python -m pip check
```

安全审查建议：

```bash
pip-audit
```

对 LLM/Agent 相关依赖的额外要求：

- 不自动升级到新主版本。
- 升级后抽样验证 20 条历史输入。
- 确认不会绕过人工审批和自动入库门控。
- 记录模型、SDK 和 prompt 变化。

### 7.4 Vue 依赖

Vue 前端使用 `package.json` 和锁文件，推荐 `pnpm-lock.yaml` 或 `package-lock.json` 固定一种。

规则：

- 提交锁文件。
- UI 组件库、路由、状态管理、表单校验属于核心依赖。
- 安全升级优先处理。
- 大版本升级单独 PR。
- 升级后至少跑：

```bash
npm run lint
npm run test
npm run build
```

如果项目使用 pnpm，则对应为：

```bash
pnpm lint
pnpm test
pnpm build
```

### 7.5 依赖升级频率

| 周期 | 任务 |
|------|------|
| 每周 | 查看安全告警，处理高危漏洞 |
| 每月 | Go/Python/Vue 常规 PATCH/MINOR 升级评估 |
| 每季度 | 核心依赖健康检查，评估 MAJOR 升级计划 |
| 发生 CVE 时 | 24 小时内评估影响，高危可走 hotfix |

## 8. 模块健康监控和异常处理

### 8.1 统一监控指标

每个模块至少关注：

- 请求量。
- 错误率。
- P95/P99 响应时间。
- 慢查询数量。
- 数据库连接数。
- 队列积压或后台任务失败数。
- 关键业务操作成功率。

统一日志字段：

| 字段 | 说明 |
|------|------|
| `timestamp` | 时间 |
| `level` | 日志级别 |
| `request_id` | 请求 ID |
| `user_id` | 当前用户，服务调用可为空 |
| `acting_user_id` | Agent 代表用户 |
| `module` | 模块名 |
| `operation` | 操作名 |
| `project_id` | 项目 ID |
| `error_code` | 错误码 |
| `duration_ms` | 耗时 |

### 8.2 模块健康指标

| 模块 | 核心健康指标 | 异常阈值建议 |
|------|--------------|--------------|
| M1 认证鉴权 | 登录成功率、token refresh 失败率、权限拒绝数量 | 登录失败突增、refresh 失败率 > 5% |
| M2 项目管理 | 项目列表耗时、成员变更失败数 | P95 > 500ms |
| M3 日志管理 | 日志创建成功率、查询耗时、附件关联失败 | 创建失败率 > 2% |
| M4 问题管理 | Issue 状态流转失败、评论失败 | 状态冲突异常突增 |
| M5 经验库 | 候选生成数、审核积压、发布失败 | 审核积压超过 7 天 |
| M6 计划与实验批次 | 任务延期数、状态更新失败 | 批次状态不一致 |
| M7 传感器 | 数据延迟、签名失败、写入失败 | 数据延迟 > 5 分钟 |
| M8 仪器控制 | 租约冲突、命令拒绝、急停事件 | 任意急停必须通知 |
| M9 Agent | 解析成功率、人工审批率、自动入库拒绝率 | 异常拒绝率突增 |
| M10 AI 问答 | 拒答率、无来源回答数、越权拦截 | 无来源回答必须为 0 |
| M11 前端 | JS 错误、页面加载耗时、API 失败提示 | 关键页面不可用 |
| M12 数据库迁移 | 迁移版本、失败次数、慢查询 | 迁移失败立即停止发布 |
| M13 部署配置 | 服务存活、磁盘、备份状态、CI 状态 | 备份失败连续 2 次 |

### 8.3 异常分级

| 等级 | 定义 | 响应 |
|------|------|------|
| P0 | 数据丢失、权限绕过、仪器危险操作、系统整体不可用 | 立即处理，必要时回滚或停用模块 |
| P1 | 核心功能不可用，例如登录、日志写入、Issue 查询失败 | 当日修复或回滚 |
| P2 | 单个模块部分功能异常，有替代路径 | 1-3 个工作日修复 |
| P3 | 文案、样式、小体验问题、低风险性能问题 | 排入常规迭代 |

### 8.4 异常处理流程

1. 确认影响范围：用户、模块、数据、时间段。
2. 指定 incident owner，避免多人同时改不同方向。
3. 判断是否需要止血：回滚、关闭 feature flag、暂停 Agent 自动入库、停用仪器控制。
4. 保留现场：日志、请求 ID、错误截图、数据库版本、部署版本。
5. 修复并验证。
6. 补充测试、监控或文档。
7. 24 小时内记录事故复盘，P0/P1 必须复盘。

### 8.5 高风险模块特殊处理

仪器控制：

- 任意未预期命令执行、租约绕过、急停事件都按 P0/P1 处理。
- 可立即停用 M8 写接口，只保留只读状态。
- 恢复前必须复核白名单、租约和审计日志。

Agent 自动入库：

- 出现错误批量写入时，先暂停自动入库，保留人工候选。
- 回滚数据前先导出受影响记录。
- 修复后用历史输入回放验证。

AI 问答：

- 出现越权引用或无来源回答时，先关闭相关知识源或问答入口。
- 修复后必须验证权限过滤和来源引用。

## 9. 定期维护任务

### 9.1 每日任务

- 查看生产服务存活状态。
- 查看 P0/P1 告警。
- 检查前一晚数据库备份是否成功。
- 查看登录、日志写入、Issue 查询、Agent 队列是否异常。
- 检查磁盘空间，尤其是附件和备份目录。
- 处理用户当天反馈的阻塞问题。

建议耗时：10-15 分钟，由值班成员执行。

- 轮流查看 13 个模块错误日志和慢接口。
- 处理安全依赖告警。
- 清理失败任务、过期临时文件、无引用附件。
- 检查 Agent 审核积压和错误样本。

- 更新本周发布计划和风险项。
- 检查测试环境与生产环境配置差异。

建议耗时：30-60 分钟，周固定时间执行。

- 依赖升级评估：Go、Python、Vue。

## 10. 回滚策略

### 10.1 回滚原则

- 先止血，再定位根因。
- 能回滚代码就不要先手工改数据。
- 数据库回滚必须谨慎，避免覆盖发布后产生的新数据。
- 回滚操作必须记录执行人、时间、版本、原因和结果。
- P0/P1 回滚不等待完整复盘，但回滚后必须复盘。

### 10.2 回滚类型

| 类型 | 适用场景 | 操作 |
|------|----------|------|
| 应用回滚 | 新版本代码导致错误，数据库兼容 | 切回上一镜像或上一 tag |
| 配置回滚 | 环境变量、开关、限流配置错误 | 恢复上一配置并重启服务 |
| Feature flag 关闭 | 新功能局部异常 | 关闭开关，保留版本 |
| API 兼容回滚 | 新 API 影响客户端 | 恢复旧行为或启用旧版本路径 |
| 数据迁移回滚 | 迁移导致 schema 或数据错误 | 执行 down 或从备份恢复 |
| 数据修复回滚 | 少量错误数据 | 基于审计日志写补偿脚本 |

### 10.3 发布前必须具备的回滚信息

每次发布说明中必须包含：

- 上一个稳定版本号。
- 当前版本是否可以只回滚应用。
- 是否包含数据库迁移。
- 数据库迁移是否可逆。
- 是否需要恢复备份。
- 是否有 feature flag 可关闭。
- 回滚预计耗时。
- 回滚负责人和备份人。

### 10.4 数据库回滚决策

优先级从高到低：

1. 如果新代码可兼容旧数据，优先只回滚应用。
2. 如果 schema 新增字段或表，一般不立即执行 `down`，保留新字段，回滚应用。
3. 如果迁移破坏旧代码运行，且 `down` 不丢数据，可以执行 `down`。
4. 如果迁移写坏数据，先导出受影响数据，再用补偿脚本修复。
5. 如果大面积数据损坏，进入备份恢复流程。

不要在没有备份和确认影响范围的情况下执行 destructive rollback。

### 10.5 回滚后检查

回滚后必须确认：

- 服务版本已回到预期版本。
- 数据库迁移版本符合预期。
- 登录、日志写入、Issue 查询等核心路径正常。
- 错误率恢复。
- 没有新的数据不一致。
- 用户需要补录或重试的操作已经说明。

## 11. 向后兼容承诺

### 11.1 默认兼容承诺

在同一个 MAJOR 版本内，系统承诺：

- 已发布 API 路径不删除。
- 已发布请求字段继续接受。
- 已发布响应字段不改名、不改变含义。
- 已发布错误码不改名。
- 已发布数据库迁移不重写。
- 已保存的历史日志、Issue、附件、实验数据可继续读取。
- 前端当前生产版本可调用当前后端 PATCH/MINOR 版本。
- Agent 当前生产版本可调用当前后端 PATCH/MINOR 版本。

### 11.2 允许 breaking change 的情况

只有以下情况可以 breaking change：

- 修复安全漏洞，例如权限绕过、越权数据访问、token 机制缺陷。
- 防止仪器危险操作或错误控制路径。
- 数据模型根本错误，继续兼容会导致数据污染或无法维护。
- 上游核心依赖或平台停止支持，继续旧接口存在安全或运行风险。
- API 已完成废弃流程，超过承诺兼容期。
- 项目进入新的 MAJOR 版本，并提供迁移说明。

### 11.3 breaking change 要求

breaking change 必须满足：

- 明确写入发布说明。
- 提供迁移文档或自动迁移脚本。
- 至少提前一个 MINOR 版本标记废弃；安全紧急修复除外。
- 测试环境提前验证。
- 前端、Agent、脚本调用方同步修改。
- 有回滚或止血方案。
- 由至少 2 人 review。

安全紧急 breaking change 可以先发布，但发布后 24 小时内必须补齐说明、测试和复盘。

### 11.4 不视为 breaking change 的变化

以下变化不视为 breaking change：

- 修复原本错误的权限放行。
- 修复明显错误的返回值。
- 增加更严格但合理的输入校验，且原输入本来不符合文档。
- 增加限流、审计和安全日志。
- 新增可选字段。
- 新增 endpoint 或新页面。

但如果这些变化会影响现有用户工作流，仍应写入发布说明。

## 12. 文档和记录要求

以下变更必须同步文档：

| 变更 | 必须更新 |
|------|----------|
| API 新增、修改、废弃 | `api-contract.md` |
| 权限规则变化 | `permission-audit.md` |
| 项目、日志、实验批次模型变化 | `project-design.md` |

## 13. 最小执行清单

如果团队时间紧，至少坚持以下底线：

- 每个模块明确主责和备份。
- 数据库变更只走迁移文件，并且发布前备份。
- API 变化先改文档，再改实现。
- 发布使用语义化版本和 tag。
- 高风险模块至少 2 人 review。
- 每日看备份和 P0/P1 告警。
- 每月做一次依赖和权限检查。
- 每季度做一次备份恢复演练。
- breaking change 必须有迁移说明和回滚方案。

这套策略可以先轻量执行，再随着系统上线规模和模块复杂度逐步加严。对于 3-4 人团队，流程的重点是减少隐性风险，而不是增加会议和表格。
