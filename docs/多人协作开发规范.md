# 多人协作开发规范

> 3-4 人协作开发实验室日志系统。每个模块独立可测、独立可验，模块间通过 API 契约解耦。

## 三、独立验证标准

每个模块交付前必须通过：

| 验证项 | 命令/方式 | 谁做 |
|--------|----------|------|
| 单元测试 | `go test ./...` | 开发者 |
| API 契约测试 | 对照 api-contract.md 逐端点 curl | 开发者 |
| 独立跑通 | 只启动本模块 + 测试 DB，不依赖其他 Go 模块 | 开发者 |
| Reviewer 环境测试 | 切换分支 → `go test` → 手动 curl | 审核者 |

示例——日志管理模块交付：

```bash
# 开发者本地
cd go-server/logs/
go test ./...                    # 单元测试全绿
# 启动测试 DB
docker run -d --name test-pg -e POSTGRES_DB=test postgres:16
# 运行迁移
psql test < migrations/001_init.sql
# 独立启动本模块
go run .                         # 只有日志模块，不需要 issues/plans
# 逐端点验证
curl -X POST /api/logs/report -d '{...}'
curl GET /api/logs/report/2026-07-13
```

## 五、Git 分支策略

```
main ───────────────────────────────────────────→  受保护，只通过 PR 合并
  │
  ├── develop ──────────────────────────────────→  日常集成
  │     │
  │     ├── feat/log-module    (开发者 A)
  │     ├── feat/issue-module  (开发者 B)
  │     ├── feat/web-ui        (开发者 C)
  │     ├── feat/py-agent      (开发者 D)
  │     └── feat/xxx
  │
  └── release/v1.0  ────────────────────────────→  发布分支
```

### 规则

- `main`：受保护分支，只能通过 PR 从 `develop` 合并。**每次合并触发 CI 全量测试。**
- `develop`：日常开发集成。所有 feature 分支先合并到 develop。
- `feat/<module>`：每个模块独立分支。一个开发者一个分支。
- **每天 push 一次**（至少），不要攒一周才推。
- 分支命名：`feat/<模块名>` `fix/<问题>` `docs/<内容>`
- 一个分支只做一个模块的事，不做跨模块改动。

## 六、代码审核流程

```
开发者 A 完成 feat/log-module
    │
    ├── 1. 本地自测：go test + 独立验证通过
    │
    ├── 2. push → GitHub
    │
    ├── 3. 开 PR：feat/log-module → develop
    │      描述：改了哪些文件、新增哪些 API、如何验证
    │
    ├── 4. 自动检查（GitHub Actions）：
    │      - go build 通过
    │      - go test 全绿
    │      - go vet 无警告
    │      - 代码格式 gofmt 一致
    │
    ├── 5. 人工 Review（至少 1 人，不能是自己）：
    │      - 对照 api-contract.md 检查接口符合契约
    │      - 检查权限中间件是否正确应用
    │      - 检查错误处理和边界情况
    │      - 在本地切分支跑一遍验证
    │
    └── 6. 审核通过 → 合并到 develop
```

**审核检查清单：**

- [ ] API 端点与 api-contract.md 一致
- [ ] 权限中间件已应用（对照 permission-audit.md）
- [ ] 敏感操作有审计日志
- [ ] 错误返回统一格式
- [ ] 没有硬编码密码/Token
- [ ] 模块没有跨模块直接调对方的 DB 表
- [ ] 测试覆盖正常路径 + 异常路径

## 七、模块合并流程

```
feat/log-module    ──PR──→  develop
feat/issue-module  ──PR──→  develop
feat/auth          ──PR──→  develop
                            │
                        集成测试
                      docker-compose up
                      curl 全量 API 回归
                            │
                      每周合并到 main
                      → v0.1, v0.2, v0.3...
```

**合并前必须：**
- 所有模块的 feature 分支已合并到 develop
- `docker-compose up` 全栈启动成功
- `curl` 遍历 api-contract.md 所有端点 200 OK
- 前端能成功调后端拿到数据

|------|------|----------|------|
| 单元测试 | `go test` | 每个函数/方法的输入输出 | 各模块开发者 |

## 九、环境与工具

| 工具 | 用途 | 安装 |
|------|------|------|
| Go 1.22+ | 后端 | `apt install golang` |

| GitHub Actions | CI 自动检查 | `.github/workflows/ci.yml` |

### CI 配置（`.github/workflows/ci.yml`）

```yaml
name: CI
on: [push, pull_request]
jobs:
  go-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with: { go-version: '1.22' }
      - run: go test ./go-server/...
      - run: go vet ./go-server/...
  frontend-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: cd web-ui && npm ci && npm test
```

## 十、常见问题

**Q: 两个模块都改了 middleware/common 怎么办？**
先到先审——先合并的那个进 develop，后合并的那个 rebase develop 解决冲突再提 PR。

**Q: 我依赖的模块还没写好怎么办？**
按 api-contract.md 的接口 mock。比如前端开发时用 MSW 模拟后端返回 JSON，不需要等 Go 写完。

**Q: DB 迁移脚本冲突？**
每人新建迁移文件，用序号区分：`002_init.sql` `003_add_experiences.sql`。不修改已有的迁移文件。
