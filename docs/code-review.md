# 代码审核规范

## 常规审核：必须别人审

PR 必须至少 1 人 approve 才能合。你审别人的，别人审你的。

## 单人/紧急审核：Codex AI 审查

当你是唯一在线的人或紧急修复时，按以下流程：

### 步骤

```bash
# 1. 你的分支 push 后，让 Codex 审查 diff
codex exec --skip-git-repo-check --sandbox danger-full-access \
  "你是本项目的代码审查员。请审查以下 PR 的代码变更。你的审查报告将决定这个 PR 能否合并。

  审查维度：
  1. 功能正确性 —— 代码是否实现了 PR 描述的功能
  2. 安全性 —— 权限检查有无遗漏、是否有注入风险
  3. 代码规范 —— 是否符合 AGENTS.md 的编码约定
  4. 模块边界 —— 有没有跨模块直接访问对方数据库表
  5. 错误处理 —— 边界情况和异常路径是否覆盖

  审查流程：
  - 先读 AGENTS.md 了解项目规范
  - 用 git diff develop...HEAD 查看变更
  - 对每个文件逐文件审查
  - 输出结构化审查报告

  报告格式：
  ## 审查结论: [通过 / 需修改 / 拒绝]
  ### 发现的问题
  | # | 严重程度 | 文件 | 行号 | 问题 | 建议修复 |
  ### 检查清单
  - [ ] 权限中间件已应用
  - [ ] 审计日志已记录
  - [ ] API 与 api-contract.md 一致
  - [ ] 无跨模块直接访问 DB
  - [ ] 无硬编码密码/Token

  只读模式，不要修改任何文件。

  注意: --sandbox danger-full-access 是必需的——Linux bubblewrap
  沙箱默认阻止文件读取，不加此参数 Codex 无法访问仓库文件。
  审查 prompt 内已明确要求只读不修改，权限由 prompt 约束。"

# 2. 根据审查报告修复问题

# 3. 在 PR 描述中附上 Codex 审查结果
#    标注「AI 审查通过 + 自审通过」

# 4. 合并 PR
```

### Codex 审查的权限边界

Codex 审查只能提供建议，**不能自动合 PR**。最终合并按钮必须由人点击。

### 什么情况不能用 Codex 审

| 情况 | 原因 |
|------|------|
| 涉及仪器控制代码变更 | 仪器安全边界需要人类判断 |
| 涉及权限规则修改 | 权限模型变更需要全部人知晓 |
| 涉及数据库迁移 | 迁移脚本必须人类审核 |
| 涉及 frp/VPS/网络配置 | 安全基础设施变更 |

以上四类必须等另一位人类开发者审核。
