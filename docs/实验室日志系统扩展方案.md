# 实验室日志管理系统扩展方案 v5（最终版）

> **状态:** 架构设计全部完成，18 项决策全部确认。v5 已补充模块 API 契约、对象级权限、审计、备份、HTTPS、数据迁移回滚与仪器安全策略。Phase 1 实施待启动。

**目标：** 将现有单人单机 SQLite + ELOG 日志系统，扩展为支持多人接入、AI Agent 自动分类录入、具备问题管理/经验管理/计划管理/EPICS 整合/仪器控制能力的模块化实验室日志平台。

**架构原则：** 高模块隔离、故障不扩散、新模块可插拔、优先采用国产/开源轻量项目。模块间只走 HTTP API，不允许跨模块直接访问对方数据库。

| 决策项 | 结论 |
|--------|------|
| 后端语言 | **Go** (chi 路由 + 标准库 net/http) |
| AI Agent 框架 | **LightAgent** (wanxingai, Apache 2.0, Python) |
| 数据库 | **PostgreSQL** 16 |
| 部署方式 | **Docker Compose** @ Rocky Linux 内网物理机 |
| 前端 | **Vue 3 + Element Plus**，PWA 适配手机 |
| 胶水/迁移 | **Python**（现有 lab_db.py 迁移 + LLM 调用） |
| ELOG | **逐步退役**，数据一次性迁移 |
| 外网访问 | **frp + VPS** (134.175.232.67) 穿透 + Nginx HTTPS |

## 五、权限模型

### 角色权限矩阵

| 模块 | admin | member | viewer | agent |
|------|-------|--------|--------|-------|
| 日志管理 | ✅ CRUD | ✅ 录/查/编 | 👁 只读 | ✅ 创建/读 |
| 问题管理 | ✅ CRUD | ✅ 创建/评论/关 | 👁 只读 | ✅ 创建/读 |
| 经验库 | ✅ CRUD | ✅ 录/查 | 👁 只读 | ❌ |
| 计划管理 | ✅ CRUD | ✅ 查/更新状态 | 👁 只读 | ❌ |
| 仪器控制 | ✅ 全控 | ⚠️ 安全写入 | 👁 仪表盘 | ⚠️ 安全写入 |
| 用户管理 | ✅ | ❌ | ❌ | ❌ |
| 系统配置 | ✅ | ❌ | ❌ | ❌ |

### Agent 硬禁止
- DELETE 任何记录
- 修改系统配置
- 连续高速扫描 (>1次/秒持续>10秒)
- 修改 HV 参数
- 同时控制多台仪器
- 不声明用户身份的操作

### 对象级权限补充

除模块级 RBAC 外，系统必须按对象做二次授权：

| 对象 | 授权粒度 |
|------|----------|
| 项目 | 读、写日志、建 Issue、管理计划、导出报告、管理成员 |
| 仪器 | 查看、申请租约、执行 green、执行 yellow、紧急停止 |
| 日志/报告 | 读取、修改、删除、导出 |
| Issue | 读取、评论、状态流转、关闭 |
| 经验库 | 读取、创建候选、审核、发布、归档 |

Agent 代用户操作时，有效权限为 `Agent 服务账号权限 ∩ 被代表用户权限 ∩ Agent 硬限制`。完整策略见 [permission-audit.md](permission-audit.md)。

## 六、用户认证

- **方式:** 用户名 + 密码
- 管理员创建账号配初始密码，首次登录强制修改
- 密码哈希优先 Argon2id；如使用 bcrypt，cost 不低于 12
- access token 使用短期 JWT（15 分钟），refresh token 使用不透明随机串并轮换
- refresh token 如放 Cookie，必须启用 HttpOnly、Secure、SameSite 和 CSRF token
- 输错 5 次锁定 15 分钟；同 IP 多账号失败触发安全告警
- 后续可升级接入所里 LDAP

## 八、传感器数据推送

- **格式:** JSON，每分钟全量 27 标签
- **端点:** `POST /api/sensors/data`
- **标签命名:** 简化（`T1` 非 `218数据_T1`），空标签保留备用
- **规模:** ~3.9 万行/天
- **鉴权:** 每台设备独立 `device_id` 和 HMAC-SHA256 签名；签名覆盖 method、path、timestamp、body hash
- **防重放:** timestamp 允许 5 分钟偏差，请求 hash/nonce 10 分钟内不得重复
- **源限制:** 内网设备绑定源 IP 或网段，tag 列表按设备白名单校验

## 九、告警与通知

| 级别 | 通道 | 场景 |
|------|------|------|
| 🔴 紧急 | ntfy (手机推送响铃) | 设备故障、温度超限、IOC 离线 |
| 🟡 日常 | MeoW (现有通道) | 日报未填提醒、周报已生成 |

### 详细触发条件

| 告警 | 级别 | 触发条件 | 恢复条件 | 防抖 |
|------|------|----------|----------|------|
| 温度超限 | 🔴 | 任一关键温度连续 3 次超过阈值，或单次超过危险阈值 | 连续 5 次回到安全范围 | 同 tag 10 分钟内合并 |
| 湿度超限 | 🟡/🔴 | 连续 5 次超过项目配置阈值；超过危险阈值升为紧急 | 连续 5 次恢复 | 同区域 30 分钟内合并 |
| PLC/传感器离线 | 🔴 | 2 个采样周期未收到全量推送 | 下一次有效推送 | 每 15 分钟重复一次 |
| IOC 离线 | 🔴 | 心跳超过 2 分钟未更新 | 心跳恢复且 PV 可读 | 每 15 分钟重复一次 |
| 仪器命令失败 | 🟡 | yellow 命令失败、超时或状态恢复失败 | 人工确认或下一次自检通过 | 同仪器 10 分钟内合并 |
| 紧急停止 | 🔴 | 任意用户触发 emergency-stop | 人工确认恢复 | 立即发送，不合并 |
| 登录攻击 | 🔴 | 管理员账号失败、同 IP 多账号失败、refresh token 复用 | 管理员处理 | 同 IP 30 分钟内合并 |
| 日报未填 | 🟡 | 工作日 20:00 后项目成员未提交日报 | 用户补交 | 每日一次 |
| 周报生成 | 🟡 | Agent 周报候选生成完成或失败 | 人工审核/重新生成 | 每周一次 |

通知投递记录必须保存状态、通道、重试次数和最终结果。紧急告警使用 ntfy 高优先级；日常提醒使用 MeoW。

## 十一、过渡期策略

- **双写 1-2 周:** Hermes 同时写 SQLite (`lab_db.py`) 和 Go API
- **验证后:** 停 SQLite 写入，保留只读归档
- 实验室电脑传感器推送保持兼容：先同时推 .md + JSON，再逐步停 .md

### 数据迁移与回滚

迁移步骤：

1. 冻结 SQLite 写入窗口，备份 `lab_env.db`、`gas_cell.db`、图片目录和 ELOG 导出。
2. 生成迁移批次号 `migration_batch_id`，所有导入记录带源表、源 ID、源 hash。
3. 先导入用户、项目、分类、附件元数据，再导入日志、Issue、经验、传感器历史。
4. 对每张表执行行数校验、hash 抽样校验、附件路径存在性校验。
5. 开启 1-2 周双写，对比 SQLite 与 PostgreSQL 的新增记录数量和关键字段 hash。
6. 验证通过后停止 SQLite 写入，只保留只读归档。

回滚策略：

- Phase 1/2 期间 PostgreSQL 作为新系统主库前，SQLite 仍保留原写入链路，可直接切回旧脚本。
- 每个迁移批次写入 `migration_batches`，回滚时按批次软删除或物理删除新库导入数据。
- 双写期发现字段映射错误时，先停新库写入，修正迁移脚本后从原始 SQLite/ELOG 重新导入该批次。
- 切主后 30 天内保留 SQLite、ELOG 导出和迁移前 PostgreSQL base backup。
- 所有回滚、重放、批量修复必须写审计日志。

### Agent 自动录入三级策略

日报解析后不入库直发，按置信度分级处理：

| 级别 | 判断标准 | 处理 |
|------|----------|------|
| 🟢 高置信 | Agent 高置信 + Go 后端规则校验通过 + 字段完整 + 设备唯一匹配 + 数值合理 | 自动入库，通知「已自动录入 1 条，可撤销或编辑」 |
| 🟡 中置信 | 字段基本可解析，但日期/项目/设备/参数存在不确定 | 生成草稿，进入「待审核中心」，一键通过/编辑 |
| 🔴 低置信 | 无法解析、疑似幻觉、prompt injection 或危险动作 | 退回提示「没理解，请改写下」或进入「待审核」让用户补全 |

关键原则：Agent 给出候选置信度，Go 后端做规则校验和权限二次确认。用户可设置偏好（高置信自动/全部确认），并可按录入类型细分。分阶段开启：冷启动全草稿→显示「本可自动入库」但需确认→校准后开放自动。详见 [agent-auto-review.md](agent-auto-review.md)。

## 十三、实施路线图

| 阶段 | 内容 | 预计 |
|------|------|------|
| **Phase 1** 安全基础设施 | Go 框架 + PostgreSQL + Docker Compose + 用户认证 + JWT/refresh + HTTPS 内网入口 + 基础审计 + 迁移脚本 | 2 周 |
| **Phase 2** 核心模块 | 日志/问题/经验 API + 对象级权限中间件 + PostgreSQL 备份/WAL + Vue 界面 + 通知基础能力 | 3 周 |
| **Phase 3** AI Agent | LightAgent 日报解析 + 问题提取 + 周报生成 | 2 周 |
| **Phase 4** 扩展模块 | 传感器接收 + 仪器控制 + EPICS Archiver + 计划管理 | 2 周 |
| **Phase 5** 生产化收尾 | frp/VPS 加固 + ntfy 完整规则 + 备份恢复演练 + 文档验收 + ELOG 退役 | 1 周 |

### PostgreSQL 备份策略

- 每日凌晨执行 `pg_dump -Fc` 逻辑备份，保留 30 天。
- 每周执行一次 base backup，配合 WAL 归档支持时间点恢复。
- WAL 归档目录独立磁盘或 NAS，同步到离线备份位置。
- 保留策略：每日逻辑备份 30 天、每周 base backup 12 周、每月归档 12 个月。
- 每月至少做一次恢复演练，恢复到临时库并校验核心表行数、最近日志、附件引用。
- 备份文件加密存储，密钥不放在同一台主机；备份、恢复、删除都写审计。
- 备份失败连续 2 次触发紧急告警。

### frp/VPS 安全加固

- VPS Nginx 只暴露 80/443；frps 管理端口不对公网开放。
- frp token 使用高强度随机值，定期轮换，配置文件权限限制为服务用户可读。
- Nginx 强制 HTTPS，启用 HSTS；证书使用 acme 自动续期并监控到期时间。
- 外网入口只反代 Web/API 必要路径，不暴露 PostgreSQL、Agent、仪器控制、EPICS、frps dashboard。
- 对 `/api/` 启用速率限制、请求体大小限制和基础 WAF 规则。
- 如访问来源固定，优先在 VPS 防火墙限制源 IP；移动访问无法固定时至少限制管理后台入口。
- SSH 禁用密码登录，只允许密钥；禁用 root 直登；fail2ban 或等价机制拦截暴力破解。
- VPS 与内网服务之间只允许 frp 隧道必要端口，内网服务仍执行 JWT、CSRF、对象级权限校验。
- 证书续期失败、frp 连接断开、异常 5xx 激增触发通知。
