# 实验室日志管理系统设计方案（Codex 独立版）

> 面向对象：实验室研究者、课题负责人、日常使用人员  
> 设计目标：把现有 SQLite + ELOG 的单人日志工具，升级为多人协作、可追溯、可扩展、带 AI 辅助和仪器/传感器接入能力的实验室管理平台。

## 1. 项目背景与目标

实验室目前的日志系统偏向“个人记录”：数据存在本地 SQLite，ELOG 用于展示或同步。这种方式对单人工作足够，但当实验室进入多人协作后，会遇到几个明显问题：

- 日志由不同成员产生，缺少统一权限、统一分类和统一检索。
- 重要问题、解决过程、经验沉淀容易散落在聊天记录、图片、个人笔记中。
- 仪器测试、传感器状态、EPICS/IOC 状态与日志没有自然关联。
- 需要外网访问时，旧系统缺少完整的认证、审计和安全边界。
- AI 可以帮助整理日志，但如果不加权限和审批，容易产生错误记录或安全风险。

本系统的目标不是简单做一个“网页日志本”，而是建设一个完整的实验室工作台：

- 支持多人录入、查询、编辑、评论和归档实验日志。
- 以项目为核心组织日志、问题、计划、经验、实验数据和报告；日报保留为“某人某天”的入口和回顾容器。
- 把项目日志、问题、计划、经验、仪器数据、传感器数据关联起来，并能按实验批次追溯。
- 用 AI Agent 辅助解析日报、生成周报、提取可复用经验；低风险普通日报可在双门控通过后自动入库，关键动作保留人工确认。
- 提供只读 AI 问答助手，帮助查询项目进展、相似历史问题、未解决风险和经验库内容。
- 允许安全地接入 Keysight E5063A、Hioki IM3536 等仪器，禁止任意远程命令。
- 支持手机和电脑使用，手机负责快速录入，电脑负责管理和分析。
- 所有重要操作可审计、可追溯、可回滚。

我的核心判断：这个项目的复杂度主要不在“写一个日志页面”，而在于项目化组织、多人权限、仪器安全、历史数据迁移和 AI 结果可信度。因此系统设计必须先把边界建牢，再逐步增加智能化和自动化。

## 2. 技术选型与理由

### Go：主后端与 API 网关

建议使用 Go 作为主后端，配合 `net/http` 或 chi 路由。

理由：

- Go 部署简单，编译成单个二进制，适合实验室内网物理机长期运行。
- 并发处理能力好，适合同时处理网页请求、设备推送、Agent 回调、通知事件。
- 类型明确，接口稳定，适合作为权限、审计、API 契约的中心。
- 标准库足够强，减少运行时依赖，降低维护成本。

我建议 Go 后端承担“可信控制面”：认证、权限、审计、业务 API、数据写入都必须经过 Go 服务。Python、Agent、仪器服务都不能绕过它直接写数据库。

### Python：AI、迁移、OCR、EPICS 胶水层

Python 适合承担非核心控制面的工作：

- 复用已有 `lab_db.py` 和 SQLite 迁移经验。
- 调用 LLM、OCR、LightAgent、EPICS CA/PVA 生态更方便。
- 适合写数据清洗、报告生成、历史导入、一次性维护脚本。

理由是 Python 在科学计算和实验室自动化生态中更自然，但不应让 Python 脚本直接成为系统安全边界。Python 服务应通过 Go API 操作数据。

### Vue 3 + Element Plus：前端

建议采用 Vue 3 + Element Plus，并做成 PWA。

理由：

- Vue 学习成本低，实验室后续维护人员更容易接手。
- Element Plus 的表格、表单、弹窗、菜单、日期选择等组件适合管理系统。
- PWA 可以在手机上接近 App 使用，不需要一开始开发原生 Android/iOS。
- 单一前端代码库能同时覆盖手机和桌面，降低开发量。

### PostgreSQL：主数据库

建议从 SQLite 升级为 PostgreSQL 16。

理由：

- 多人并发写入、权限查询、复杂筛选、审计日志、传感器历史数据都更适合 PostgreSQL。
- 支持事务、外键、索引、JSONB、全文检索、分区表，后期扩展空间大。
- 备份、WAL 归档、时间点恢复比 SQLite 更适合生产环境。

SQLite 适合单人单机，继续作为历史只读归档可以，但不适合作为多人平台主库。

### LightAgent：AI Agent 框架

建议使用 LightAgent 承担日报解析、周报生成、经验候选提取。

理由：

- 与 Python 生态适配，便于快速接入大模型和工具调用。
- Apache 2.0 许可证更利于内部长期使用。
- Agent 可以作为独立服务演进，不影响主后端稳定性。

但我有一个明确保留意见：Agent 只能生成候选结果和低风险草稿，不能直接拥有“系统管理员”或“仪器操作者”的能力。Agent 的有效权限应始终是“Agent 服务账号权限、被代表用户权限、Agent 硬限制”的交集。

自动入库也不能只看 Agent 自评置信度。正式策略应是“Agent 候选置信度 + Go 后端规则置信度 + 用户偏好”双门控：只有普通文本日报、项目和字段唯一、权限通过、无重复冲突、用户开启对应偏好时，才允许高置信自动提交。OCR、指定设备或高风险对象默认进入人工确认。

### Docker Compose + Rocky Linux

建议第一阶段使用 Docker Compose 部署在 Rocky Linux 内网物理机上。

理由：

- 实验室规模下不需要一开始上 Kubernetes。
- Compose 便于固定 PostgreSQL、Go API、Agent、前端、通知等服务版本。
- Rocky Linux 稳定，适合长期内网服务器。

## 3. 系统架构

系统采用“一个可信入口，多个独立服务”的架构。

```text
手机/电脑浏览器
        |
        | HTTPS
        v
VPS Nginx + frp server（外网入口）
        |
        | frp 隧道
        v
Rocky Linux 内网服务器
        |
        +-- Go API 网关 / 业务后端
        |       +-- 认证、权限、审计
        |       +-- 日志、Issue、经验、计划 API
        |       +-- 传感器接收 API
        |       +-- 仪器命令 API
        |       +-- AI 问答只读 API
        |
        +-- PostgreSQL
        |
        +-- Vue PWA 静态前端
        |
        +-- Python LightAgent 服务
        |
        +-- QA Orchestrator / RAG 索引服务
        |
        +-- OCR / 报告生成 / 迁移脚本
        |
        +-- 仪器控制服务
        |
        +-- EPICS/PLC 接入服务
```

### 数据流

普通日志流：

```text
用户输入自然语言或上传图片
    -> Go API 创建个人日报容器/附件
    -> LightAgent 解析候选字段
    -> 拆成 daily_report_items 或 logs，绑定项目
    -> Agent 置信度 + Go 后端规则校验 + 用户偏好
    -> 用户确认，或低风险文本日报自动提交
    -> PostgreSQL 保存项目日志
    -> 审计记录写入
```

AI 问答流：

```text
用户在项目页或全局入口提问
    -> Go API 鉴权并绑定当前用户
    -> QA Orchestrator 识别意图和上下文
    -> 调用预定义只读查询函数
    -> 权限过滤 RAG 召回日志、Issue、经验、计划
    -> LLM 生成带来源回答
    -> 审计记录问题 hash、工具、对象 ID 和结果状态
```

问题与经验流：

```text
日志或成员创建 Issue
    -> Issue 跟踪处理过程
    -> Issue 关闭后进入经验候选扫描
    -> Agent 生成经验候选
    -> 人工审核
    -> 发布到经验库
```

仪器控制流：

```text
用户申请仪器租约
    -> Go API 校验用户和仪器权限
    -> 选择白名单命令
    -> 服务端校验参数范围、对象类型、互斥锁、超时
    -> 仪器服务发送 SCPI
    -> 记录前后状态和审计
    -> 返回结果或告警
```

传感器数据流：

```text
PLC/传感器设备每分钟推送 JSON
    -> HMAC 签名校验
    -> tag 白名单校验
    -> 写入 sensor_readings
    -> 更新最新值
    -> 触发阈值告警判断
```

## 4. 模块设计与职责

### 认证与用户模块

职责：

- 用户名密码登录。
- 管理员创建账号和初始密码。
- 首次登录强制修改密码。
- JWT access token + refresh token 轮换。
- 登录失败锁定、会话撤销。

理由：多人系统必须先能确认“谁在操作”。如果身份体系薄弱，后续审计和权限都会失效。

### 项目模块

职责：

- 管理项目基本信息、成员、项目内角色和生命周期。
- 项目状态采用 `draft -> active -> completed -> archived`。
- 支持项目级 ACL，用于控制项目日志、Issue、计划、测试数据、经验候选和报告。
- 不做子项目层级，第一阶段只保留 `project + experiment_runs`。

设计重点：项目是系统的主组织维度。日志、Issue、测试数据、计划、报告都必须能回到一个主项目；跨项目内容用关联表表达，统计默认只按主项目计数。仪器权限不从项目继承，项目写权限不能自动变成仪器操作权限。

### 权限模块

职责：

- 管理角色：`admin`、`project_owner`、`member`、`viewer`、`instrument_operator`、`auditor`、`agent`、`device`。
- 管理对象级 ACL：项目、日志、Issue、经验、计划、仪器、报告、附件、审计。
- 支持显式 deny，用于临时冻结或敏感对象限制。

理由：实验室数据不是所有人都应全部可见，仪器操作也不能只靠“登录了就能控制”。模块权限解决“能不能进页面”，对象权限解决“能不能操作这条记录或这台仪器”。

### 日志模块

职责：

- 创建、查询、编辑、软删除项目日志。
- 保留 `daily_reports` 作为某人某天的个人日报容器。
- 新增 `daily_report_items` 或长期 `logs` 作为真正挂项目的工作记录。
- 支持分类、项目、作者、时间、附件、关联 Issue、关联计划、关联仪器数据和实验批次。
- 支持 Agent 解析候选，但人工可修正。

设计重点：日报不是项目归属边界。一条日报可以包含多个项目的条目，每条正式日志必须有主项目。无项目记录只允许作为草稿、待分类或公共事务记录，不能长期进入正式项目统计。

### 附件与 OCR 模块

职责：

- 上传图片、PDF、仪器截图。
- 生成缩略图、保存 sha256、文件路径、关联对象。
- OCR 提取文本，但 OCR 结果标记为“不可信输入”。

理由：图片和仪器截图在实验室很重要，但文件名、OCR 内容、图片文字都可能包含错误或恶意指令，不能直接作为 Agent 指令执行。

### 问题管理模块

职责：

- 记录实验异常、设备故障、分析疑问。
- 支持严重度、负责人、状态流转、评论。
- 与日志、计划、附件关联。

理由：日志记录“发生了什么”，Issue 记录“这个问题如何被处理”。经验库应主要从已解决 Issue 中提取，而不是从零散日志中盲目总结。

### 经验库模块

职责：

- 保存经过审核的实验经验、排查流程、注意事项。
- Agent 每周生成候选。
- 人工审核、拒绝、发布、归档。

理由：经验库应该是“可信知识”，不是大模型自动生成的备忘录。必须把候选和正式发布分开。

### AI 问答模块

职责：

- 提供只读实验数据问答入口。
- MVP 只覆盖 4 类问题：项目进展、相似历史问题、未解决风险、已发布经验查询。
- 通过预定义只读查询函数和权限过滤 RAG 获取资料。
- 回答必须带时间范围、数据来源、来源链接和不确定性说明。

设计重点：第一阶段不开放通用 Text-to-SQL。统计类问题，例如 RF 匹配成功率、传感器异常次数，等结构化 `result`、阈值、质量标记等字段补齐后再上线。

### 计划管理模块

职责：

- 管理装配、测试、实验计划。
- 支持任务、负责人、进度、里程碑、状态。
- 与日志和 Issue 关联。

理由：计划模块帮助回答“这周该做什么、卡在哪里、完成依据是什么”。

### 仪器控制模块

职责：

- 管理仪器状态、租约、互斥锁。
- 只允许执行 YAML 白名单命令。
- 根据风险级别区分 green、yellow、red。
- 记录操作前后状态、参数、结果、审批。

理由：仪器控制有物理风险，设计上必须默认保守。前端按钮和 Agent 建议都不是安全边界，真正边界必须在后端和仪器服务。

### 传感器与 EPICS 模块

职责：

- 接收 PLC/传感器 JSON 全量推送。
- 记录最新值和历史值。
- 接收 IOC 心跳。
- 后期接入 EPICS PV 归档。

理由：传感器和 EPICS 数据能为日志提供客观上下文，例如温度、湿度、真空、IOC 在线状态。

### 通知模块

职责：

- 处理紧急告警和日常提醒。
- 紧急事件走 ntfy，日常提醒走 MeoW。
- 记录投递状态、失败原因和重试。

理由：告警系统不能只“发一下消息”，还要知道是否发出、是否失败、是否需要升级。

### 审计模块

职责：

- 记录登录、权限变更、业务写入、Agent 动作、仪器命令、数据导出、备份恢复。
- 审计表 append-only，应用账号不应有 UPDATE/DELETE 权限。

理由：审计不是为了事后追责而已，更是系统出问题时定位原因的依据。

## 5. 数据库设计思路

数据库采用 PostgreSQL。表结构不追求一开始非常复杂，但需要保留扩展空间。

### 用户与权限

- `users`：用户账号、姓名、状态、密码 hash、首次登录标记。
- `roles`：角色定义。
- `user_roles`：用户与角色关系。
- `sessions`：refresh token hash、设备信息、过期时间。
- `acl_entries`：对象级权限，记录对象、主体、动作、授权人、过期时间。
- `service_accounts`：Agent、仪器服务等服务身份。
- `devices`：PLC、IOC 等设备身份和 HMAC key 版本。

### 项目与日志

- `projects`：项目基本信息、短码、负责人、生命周期状态。
- `project_members`：项目成员和项目内角色。
- `daily_reports`：个人每日容器，按 `report_date + author_id` 唯一，不直接放 `project_id`。
- `daily_report_items`：日报下的项目化条目，迁移阶段承载项目、分类、正文、来源和置信度。
- `logs`：长期正式日志表，记录正文、发生时间、作者、主项目、分类、来源、状态。
- `daily_report_log_links`：如果采用长期 `logs` 方案，用于把日报和日志关联起来。
- `log_links`：日志与 Issue、仪器数据、计划任务的关联。
- `experiment_runs`：一次降温、一次束流实验、一次 RF 匹配扫描等轻量实验批次。
- `attachments`：附件元数据、路径、sha256、mime、缩略图路径。
- `attachment_links`：附件绑定到日志、Issue、经验等对象。

项目状态固定为 `draft`、`active`、`completed`、`archived`。`draft` 主要用于补齐配置和成员；`active` 正常写入；`completed` 允许收尾总结、补录和经验发布；`archived` 默认只读。第一阶段不做子项目层级，跨日或跨任务归集由 `experiment_runs` 和计划任务承担。

### 问题、经验、计划

- `issues`：问题标题、描述、主项目、状态、严重度、负责人。
- `issue_project_links`：跨项目影响关系，权限以主项目为准。
- `issue_comments`：问题评论。
- `issue_transitions`：状态流转记录。
- `experiences`：正式经验，可挂主项目，少量全局经验必须审核和脱敏。
- `experience_project_links`：经验跨项目适用关系。
- `experience_candidates`：Agent 生成的候选经验。
- `experience_reviews`：审核记录。
- `plans`：项目计划。
- `plan_tasks`：任务、进度、负责人、时间。
- `plan_milestones`：里程碑。

### 仪器与实验数据

- `instruments`：仪器基本信息、状态、IP、类型。
- `instrument_permissions`：独立仪器权限，不从项目成员关系继承。
- `instrument_leases`：仪器占用租约。
- `instrument_commands`：命令记录、白名单版本、参数 hash、结果摘要。
- `instrument_measurements`：测试结果摘要或数据文件引用。
- `instrument_state_snapshots`：操作前后状态快照。
- `test_data` / `rf_matching_circuits` / `assembly_process` / `cryo_runs`：应补齐 `project_id`，可选关联 `experiment_run_id`。

### 传感器与 EPICS

- `sensor_devices`：设备配置、允许 tag、源 IP。
- `sensor_readings`：历史读数，建议按时间分区。
- `sensor_latest`：每个 tag 最新值，便于仪表盘快速读取。
- `epics_ioc_heartbeats`：IOC 心跳。
- `epics_pv_metadata`：PV 元数据，后期扩展。
- `epics_pv_samples`：PV 历史归档，后期视数据量决定是否独立存储。

### Agent、通知、审计、迁移

- `agent_tasks`：Agent 任务状态。
- `agent_candidate_actions`：候选动作。
- `agent_candidate_reviews`：人工审核和规则判定记录。
- `agent_user_preferences`：用户自动入库偏好，例如文本日报自动、OCR 不自动、指定设备必须确认。
- `agent_prompt_versions`：Prompt 和工具 schema 版本。
- `qa_sessions`：短期问答会话元数据。
- `qa_audit_events`：问题 hash、意图、工具、对象 ID、结果状态和耗时，不默认保存完整敏感问答。
- `notification_events`：告警事件。
- `notification_deliveries`：通知投递记录。
- `audit_events`：审计事件。
- `migration_batches`：迁移批次。
- `idempotency_keys`：写接口幂等记录。

我的建议：传感器历史数据如果增长很快，应尽早使用按月或按周分区，并提前设计保留策略。不要等数据量变成几千万行后再补救。

## 6. API 设计原则

API 应采用 `/api/v1/{module}/{resource}` 的 REST 风格。

基本原则：

- 所有业务写接口必须带 `Idempotency-Key`，避免重复提交。
- 所有响应带 `request_id`，便于查日志。
- 所有错误统一格式，前端和 Agent 都能处理。
- 时间统一使用 RFC3339，带时区。
- 列表接口使用游标分页，不建议用大 offset。
- 删除默认软删除，真正物理删除只用于管理员维护和合规清理。
- API 字段只增不删，避免前端、Agent、脚本同时被破坏。
- Agent、设备、用户使用不同认证方式，不能混用。

模块间通信原则：

- 前端只访问 Go API。
- Agent 只访问 Go API，不直接写 PostgreSQL。
- 仪器服务只接受白名单命令，不接受任意 SCPI。
- 传感器设备只调用设备推送接口，不使用普通用户 token。
- Go API 网关注入鉴权、权限、审计上下文。

我的不同意见：文档中强调“模块之间只走 HTTP API，不直接访问对方数据库”是对的，但在一个小团队项目里，不建议第一阶段把每个模块都拆成独立数据库或独立微服务。可以先做成一个 Go 单体后端，内部按模块分包，外部契约仍保持 HTTP/API 边界。这样更容易调试和上线，后期再拆。

## 7. 前端设计：PWA 双端适配

前端采用单一 Vue 代码库，同时适配手机和桌面。

### 手机端

手机端应围绕“快速操作”设计：

- 今日日报快速录入，按项目拆分为多条记录。
- 拍照上传。
- AI 解析候选确认。
- 底部导航进入 AI 问答对话面板。
- 今日仪表盘：温湿度、真空、设备在线、告警。
- 最新日志卡片流。
- 简化版 Issue 创建和评论。
- 仪器状态查看和安全命令发起。

手机端不应塞入过多管理表格。它更适合现场记录、拍照、确认和查看告警。

移动端 AI 问答不使用悬浮球，避免遮挡录入和仪器状态操作。推荐放在底部导航中，打开后使用全屏或近全屏对话面板，并自动带入当前项目、Issue 或测点上下文。

### 桌面端

桌面端应围绕“管理和分析”设计：

- 项目列表和项目仪表盘：最近动态、实验批次、计划、Issue、关键指标。
- 日志表格：筛选、搜索、排序、导出。
- Issue 看板：按状态、负责人、严重度查看。
- 经验库审核：候选、来源、差异、批准/拒绝。
- 计划甘特图或任务看板。
- 仪器控制台：租约、命令、结果、审计。
- 传感器曲线和 IOC 状态。
- AI 问答右侧侧边栏：在项目、Issue、经验、传感器页面内保留上下文提问。
- 用户、权限、审计、备份状态管理。

桌面端 AI 问答采用右侧侧边栏，不作为全屏主页面起步。项目页、Issue 详情页、经验库和传感器页提供“问这个项目/问题/测点”的入口，回答下方必须展示可点击来源。

### PWA 离线能力

建议只做有限离线能力：

- 手机端可缓存最近页面和草稿。
- 网络恢复后由用户确认再提交。
- 仪器控制、权限修改、经验发布、数据导出不允许离线排队执行。

疑虑：如果离线队列自动重放写操作，容易造成重复日志或过期仪器命令。因此离线能力应从“草稿保存”开始，不要一开始做复杂自动同步。

## 8. AI Agent 设计

AI Agent 的定位是“助手”，不是“管理员”。

### 日报解析

输入：

- 用户自然语言。
- 图片 OCR 文本。
- 相关项目候选。
- 用户身份和可用权限摘要。

输出：

- 日期、项目、分类、正文、可能的标签。
- `agent_confidence`：Agent 自评置信度。
- 字段级来源证据，例如日期、项目、设备、单位来自哪段文本或哪个已知别名。
- 是否建议自动提交。

正式处理由 Go 后端二次判断：

- `rule_confidence`：项目、日期、人员、设备、单位、数值范围、重复冲突、权限和生命周期规则校验后的结果。
- `final_decision`：`auto_committed`、`draft_pending_review`、`needs_clarification` 或 `rejected`。

高置信自动入库必须同时满足 Agent 高置信、后端规则高置信、用户偏好允许、记录类型是普通文本日报、所有关键字段可追溯。OCR、图片、指定设备相关记录、Issue、经验、仪器命令、权限、导出和报告发布都默认人工确认。

自动入库分阶段开启：

1. 冷启动阶段：全部生成草稿。
2. 观察阶段：系统标记“本可自动入库”，但仍需确认。
3. 自动阶段：用户手动开启细分偏好后，高置信普通日志可自动提交。

用户偏好不能只有“自动/不自动”一个开关。至少应支持文本日报自动、OCR 永不自动、指定项目允许自动、指定设备相关记录必须确认。

### 周报生成

每周按项目生成：

- 本周完成事项。
- 关键实验结果。
- 未解决问题。
- 下周计划建议。
- 需要关注的风险。

周报来源应可追溯到日志、Issue、计划任务和传感器/仪器数据。报告中每个重要结论最好能点回原始记录。

### 经验提取

经验提取建议从“已关闭 Issue + 关联日志 + 评论”中产生，而不是从全部日志中泛泛总结。

候选经验应包括：

- 问题背景。
- 现象。
- 排查步骤。
- 最终原因。
- 解决方法。
- 适用范围。
- 不适用或注意事项。
- 来源链接。

发布前必须人工审核。Agent 不得直接发布正式经验。

### Prompt Injection 防护

所有日志正文、OCR 文本、评论、历史迁移文本都属于不可信输入。Agent payload 应明确区分：

- `trusted_context`：系统给出的权限、任务、允许动作。
- `untrusted_inputs`：用户文本、OCR、历史内容。

Agent 不应执行不可信输入中的“忽略之前指令”“调用工具”“删除数据”等内容。所有工具调用都必须经过 Go API 的权限和 schema 校验。

我建议建立一组专门的 prompt injection 回归样例，每次改 Prompt 或工具 schema 都跑一次。这是 AI 模块能否长期可靠的关键。

## 9. AI 问答助手设计

AI 问答助手是只读分析入口，和快速录入/Agent 写入流程分开。它不能执行写操作、不能控制仪器、不能导出批量数据，也不能确认用户无权项目是否存在。

第一版 MVP 只做 4 类问题：

- 项目最近进展：基于项目日志、计划、Issue 和实验批次。
- 相似历史问题：基于已解决 Issue、日志和经验库。
- 未解决风险：基于 open/in_progress Issue、阻塞计划和近期异常记录。
- 经验查询：优先引用已发布经验，再补充有权限的历史解决记录。

技术方案采用预定义只读查询函数 + 权限过滤 RAG：

- 结构化资料通过 `projects.summary`、`logs.search`、`issues.search`、`issues.similar`、`plans.status`、`experiences.search` 等工具获取。
- 非结构化资料通过 RAG 召回，但每个 chunk 必须带 `project_id`、对象类型、对象 ID、时间和权限标签。
- 检索前先确定当前用户可读项目集合，不能先召回再让 LLM 判断权限。
- 回答必须包含时间范围、数据来源类型和数量、关键结论、来源链接和不确定性。

统计类问题暂缓到结构化字段补齐后再上线。例如 RF 匹配成功率需要 `result`、`failure_reason`、`validated_at` 等字段；传感器异常需要项目阈值、质量标记和采样缺失规则。字段不足时，QA 应说明数据不足或口径不稳，而不是强行给结论。

## 10. 仪器控制与安全

仪器控制采用白名单机制。当前纳入：

- Keysight E5063A：LAN SCPI，`10.51.12.157:5025`。
- Hioki IM3536：LAN SCPI，`10.51.12.101:3500`。

命令分级：

- green：只读命令，例如识别仪器、读取状态、读取测量值。
- yellow：安全写入命令，例如设置扫频范围、设置功率、触发单次测量。
- red：危险命令，例如 reset、preset、校准，远程默认拒绝。

安全要求：

- 后端只接受命令名和结构化参数，不接受原始 SCPI。
- YAML 白名单启动时必须做 schema 校验，失败则仪器服务不启动。
- yellow 命令必须有对象级权限、有效租约、互斥锁、参数范围校验、超时。
- `set_sweep_range` 必须校验 start < stop、最大跨度、点数、IFBW 与预计时间。
- `set_power`、`set_dc_bias` 必须根据被测对象类型收窄范围。
- `trigger_single` 必须保存状态快照，执行后恢复，恢复失败要告警。
- 所有 yellow 命令记录操作前后状态和审计。
- Agent 发起的任何 yellow 命令必须人工确认。
- 紧急停止所有登录成员都能触发，触发后进入人工检查锁定状态。

我对当前方案的一个加强建议：截图命令最好不要让用户传完整路径，只让用户传一个短 label，由后端生成安全文件名。现有白名单已经有路径正则，但后端生成文件名更稳。

## 11. 传感器与 EPICS 整合

### PLC/传感器

建议第一阶段采用设备主动推送：

- 每分钟推送一次全量 JSON。
- 每台设备有独立 `device_id` 和 HMAC secret。
- 服务端校验签名、时间戳、防重放、源 IP、tag 白名单。
- 数据写入 `sensor_readings`，最新值写入 `sensor_latest`。

理由：主动推送比服务器轮询更容易穿过网络边界，也更容易从设备侧控制采样节奏。

### EPICS

建议分两步做：

第一步：

- IOC 心跳上报。
- 前端显示 IOC 在线/离线。
- 关键 PV 的最新值展示。

第二步：

- 建立 EPICS PV 元数据表。
- 对关键 PV 做归档。
- 与实验日志、告警、周报关联。

疑虑：EPICS PV 归档如果一开始就做全量，数据规模和维护复杂度可能迅速上升。建议先归档少量关键 PV，确认查询方式和保留策略后再扩大范围。

## 12. 权限与审计

权限应分层判断：

1. 身份是否有效。
2. 是否有模块权限。
3. 是否有对象权限。
4. 是否允许执行具体动作。
5. 是否触发高风险规则。

必须审计：

- 登录、登出、登录失败、账号锁定。
- 用户、角色、权限、设备密钥、服务账号变更。
- 所有业务写操作。
- 所有 Agent 代用户操作。
- 所有仪器命令、租约、紧急停止。
- 数据导出、报告生成、附件下载。
- 备份、恢复、迁移、回滚。

审计日志建议保存：

- 谁操作。
- 代表谁操作。
- 从哪里操作。
- 对哪个对象操作。
- 参数摘要和 hash。
- 结果、错误码、耗时。
- request_id、trace_id、idempotency_key。
- Agent prompt/tool schema 版本。

重要原则：

- 审计日志不可被普通应用逻辑修改。
- 敏感字段只存 hash 或脱敏摘要。
- 仪器控制审计保留时间应长于普通日志，建议不少于 5 年。

## 13. 外网访问方案：frp

建议使用 VPS + frp + Nginx HTTPS。

拓扑：

```text
外网用户
  -> VPS Nginx 443
  -> frp server
  -> 内网 frp client
  -> Rocky Linux Go API / Vue PWA
```

安全要求：

- VPS 只开放 80/443 和必要 SSH。
- frps 管理端口不对公网开放。
- Nginx 强制 HTTPS，启用 HSTS。
- 使用 acme 自动续期证书，并监控到期。
- 外网只暴露 Web/API，不暴露 PostgreSQL、Agent、仪器服务、EPICS、frps dashboard。
- `/api/` 设置速率限制和请求体大小限制。
- SSH 禁用密码登录，禁用 root 直登。
- frp token 使用强随机值并定期轮换。
- 外网访问仍必须走 JWT、CSRF、对象权限，不因为来自 frp 就信任。

我的疑虑：frp 适合快速稳定地解决外网访问，但它把内网服务暴露到了公网路径上。上线前必须做最小暴露和备份恢复演练，不能只验证“能打开页面”。

## 14. 实施路线图

### Phase 0：准备与冻结口径

目标：明确范围，避免边做边失控。

- 确认首批用户、项目、仪器、传感器 tag。
- 确认项目生命周期、项目成员角色和默认公共事务项目。
- 确认日志分类和 Issue 状态。
- 确认 `daily_reports` 日容器与 `daily_report_items`/`logs` 的迁移取舍。
- 确认 AI 问答 MVP 只做项目进展、相似历史问题、未解决风险和经验查询。
- 确认 Agent 自动入库默认冷启动全草稿，后续观察再开启自动。
- 确认哪些数据必须迁移，哪些只保留归档。
- 确认外网访问域名、VPS、证书方案。

### Phase 1：安全基础与数据底座

目标：先把系统“站起来且可信”。

- Go API 基础框架。
- PostgreSQL schema 初版：用户、项目、项目成员、日报容器、项目日志/条目、Issue、附件、审计。
- Docker Compose。
- 用户登录、JWT、refresh token。
- 基础 RBAC、项目 ACL 和独立仪器权限。
- 审计日志。
- 备份脚本和恢复演练。
- SQLite/ELOG 迁移脚本草案。

### Phase 2：核心业务模块

目标：替代旧日志主流程。

- 项目列表、项目仪表盘和项目生命周期管理。
- 日报录入，按项目拆分为 `daily_report_items` 或正式 `logs`。
- 日志查询、编辑、附件上传。
- Issue 创建、评论、状态流转。
- 计划、实验批次 `experiment_runs` 和项目周报基础数据。
- 经验库候选和人工审核。
- Vue PWA 手机/桌面基础界面。
- 通知基础能力。
- 双写 SQLite 和 PostgreSQL 1 到 2 周。

### Phase 3：AI Agent

目标：让 AI 进入辅助流程，但不接管安全边界。

- 日报解析候选。
- 问题提取建议。
- 周报生成。
- 经验候选提取。
- Agent 任务表、候选动作、人工审批和待审核中心。
- 三级自动入库：冷启动全草稿、观察阶段标记“本可自动”、用户开启后仅普通文本日报自动。
- 双门控：Agent 候选置信度 + Go 后端规则校验。
- 用户偏好：文本日报自动、OCR 不自动、指定项目允许自动、指定设备必须确认。
- Prompt injection 回归测试。

### Phase 4：AI 问答、传感器、EPICS、仪器控制

目标：接入只读智能查询、实验现场数据和受控仪器能力。

- AI 问答桌面右侧侧边栏和移动端底部对话面板。
- 只读查询函数和权限过滤 RAG。
- MVP 四类问题：项目进展、相似历史问题、未解决风险、经验查询。
- PLC/传感器 HMAC 推送。
- 最新值仪表盘和历史曲线。
- IOC 心跳。
- 仪器租约、白名单命令、互斥锁、紧急停止。
- 仪器命令审计和通知。

统计类 AI 问答在这一阶段只做字段预留和口径设计。RF 成功率、传感器异常次数等问题等结构化字段和阈值配置稳定后再开放。

我建议把仪器控制放在传感器之后上线，因为传感器是只写数据，风险较低；仪器控制会影响真实设备，应在权限、审计、告警成熟后再开放。

### Phase 5：生产化收尾

目标：从“可用”变成“可长期运行”。

- frp/VPS 加固。
- HTTPS、HSTS、速率限制。
- ntfy/MeoW 完整告警规则。
- 备份恢复月度演练。
- 迁移验收和旧系统只读归档。
- 编写用户手册和管理员手册。

## 15. 潜在风险与缓解措施

### 历史数据迁移错误

风险：SQLite/ELOG 中字段不一致、图片路径缺失、时间格式混乱。

缓解：

- 每次迁移使用 `migration_batch_id`。
- 保存源表、源 ID、源 hash。
- 做行数校验、hash 抽样、附件存在性检查。
- 双写期间对比新旧系统新增记录。
- 切主后至少 30 天保留旧库和导出。

### 权限模型过复杂

风险：权限做得太细导致使用和管理困难。

缓解：

- 第一阶段只开放常用角色和项目权限。
- 仪器权限单独管理。
- ACL 后台提供“按项目复制权限”和“权限预览”。
- 审计员角色只读审计，不参与业务管理。

### Agent 生成错误内容

风险：AI 总结错误、把 OCR 错字当事实、受 prompt injection 影响。

缓解：

- 冷启动阶段 AI 输出默认是候选草稿。
- 高置信自动入库必须通过 Agent 置信度和 Go 后端规则双门控。
- 用户偏好细分，OCR 和指定设备可强制人工确认。
- 自动入库记录保留来源、规则命中、模型版本和可撤销入口。
- 重要结论必须有来源链接。
- 经验发布必须人工审核。
- 工具调用经过 schema 和权限校验。
- 建立 prompt injection 测试集。

### AI 问答越权或口径不稳

风险：QA 泄露用户无权项目、把 RAG 结果当事实、在结构化字段不足时给出错误统计。

缓解：

- QA 只读，不能写业务表、不能控制仪器、不能批量导出。
- 检索前按当前用户项目权限过滤 RAG chunk。
- MVP 只做项目进展、相似历史问题、未解决风险和经验查询。
- 统计类问题等 `result`、阈值、质量标记等结构化字段补齐后再上线。
- 回答必须带来源；没有来源时说明数据不足。

### 仪器误操作

风险：远程命令导致仪器状态异常、扫频范围过大、功率设置不当。

缓解：

- 原始 SCPI 永不开放。
- yellow 命令必须租约、互斥锁、范围校验。
- red 命令默认拒绝。
- 执行前后状态快照和恢复。
- 紧急停止入口常驻。
- Agent 的仪器写操作必须人工确认。

### 传感器数据量增长

风险：每分钟全量数据长期积累后查询变慢、备份变大。

缓解：

- 按时间分区。
- `sensor_latest` 单独存最新值。
- 历史曲线做降采样。
- 明确保留策略，例如原始数据保留 1 年，降采样数据保留更久。

### 外网暴露风险

风险：frp/VPS 配置不当导致管理端口、内网服务或数据库暴露。

缓解：

- 只反代 Web/API。
- 防火墙限制端口。
- Nginx 限流。
- 强制 HTTPS。
- SSH 加固。
- 定期检查公网开放端口。

### 单机部署故障

风险：Rocky Linux 物理机故障导致系统不可用。

缓解：

- 每日逻辑备份、WAL 归档、每周 base backup。
- 备份放到不同磁盘或 NAS。
- 每月恢复演练。
- 文档记录重建步骤。

## 16. 当前方案需要改进或加强的地方

### 1. 建议第一阶段采用“模块化单体”，不要过早微服务化

已有方案强调模块隔离是正确的，但实验室系统初期更需要稳定交付。建议 Go 后端内部模块化，数据库表按模块清晰划分，对外 API 契约保持独立。等某个模块确实需要独立扩容或独立部署时再拆服务。

### 2. 需要明确传感器数据保留和降采样策略

当前方案已估算每天约 3.9 万行，但还需要明确：

- 原始数据保留多久。
- 曲线查询默认粒度。
- 是否生成 5 分钟、1 小时、1 天聚合表。
- 备份是否包含全部历史传感器原始数据。

否则一年后数据库会明显变重。

### 3. 仪器控制应晚于日志、权限、审计上线

仪器控制很有价值，但物理风险最高。建议先让日志、Issue、权限、审计稳定运行，再开放仪器 yellow 命令。早期可以先做只读 green 命令和状态查看。

### 4. Agent 自动写入范围应采用双门控和分阶段开启

我建议：

- 冷启动阶段全部生成草稿。
- 观察阶段只提示“本可自动入库”，用真实数据校准规则。
- 自动阶段只允许普通文本日报在双门控通过后自动入库。
- 用户偏好细分到来源、项目和设备。
- Issue、经验、仪器命令、报告导出都应默认人工确认。

这会减少 AI 初期误操作带来的信任损耗。

### 5. 需要补充测试策略

现有方案对功能设计较完整，但测试策略还可以加强：

- API 契约测试。
- 权限矩阵测试。
- 仪器白名单 schema 测试。
- 传感器 HMAC 防重放测试。
- Agent prompt injection 回归测试。
- 数据迁移校验测试。
- 备份恢复演练测试。

### 6. 需要补充运维界面

建议增加管理员运维页：

- PostgreSQL 备份状态。
- 最近一次恢复演练记录。
- frp 连接状态。
- 证书到期时间。
- Agent 队列状态。
- 传感器最后推送时间。
- IOC 在线状态。
- 仪器 needs_check 状态。

这些信息能让非软件工程师也知道系统是否健康。

### 7. 需要明确“项目”作为核心组织单位

日志、Issue、经验、计划、报告都建议围绕项目组织。`daily_reports` 保留为日容器，真正进入项目统计的是 `daily_report_items` 或 `logs`。仪器权限不自动从项目继承，但仪器数据可以关联项目。这样后续做项目周报、项目归档、项目权限会更自然。

### 8. 需要定义人工审核体验

经验候选、Agent 动作、仪器高风险命令都需要人工确认。前端应提供统一的“待审核中心”，否则审核会散落在多个页面，最终没人处理。

### 9. AI 问答应先做只读 MVP

AI 问答不要一开始做通用数据库问答。第一版只做项目进展、相似历史问题、未解决风险和经验查询，采用预定义只读查询函数和权限过滤 RAG。桌面端放右侧侧边栏，移动端放底部导航对话面板；统计类问题等结构化字段补齐后再开放。

## 17. 总结

我建议把这个平台定位为“实验室工作记录与知识沉淀中心”，而不是单纯的 ELOG 替代品。

最小可行路径是：

1. 先上线用户、权限、审计、日志、附件。
2. 再上线项目、项目日志、Issue、经验候选、计划和实验批次。
3. 然后接入 AI 辅助整理和受控自动入库。
4. 再上线只读 AI 问答 MVP。
5. 最后开放传感器、EPICS 和仪器控制。

这样做的好处是每一步都有独立价值，并且不会在权限、审计、备份还不成熟时过早引入仪器控制和 AI 自动化风险。
